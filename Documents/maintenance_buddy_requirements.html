Maintenance Buddy Requirements

Team members:
Ashton Hull, Jordan Glatz, Daniel Vincent

1. Purpose

The purpose of this document is to define an outline for the requirements of Maintenance Buddy. Maintenance Buddy is a tool in the form of a mobile app that helps keep a complete maintenance log for your vehicle in one easy to access place. Users have the ability to input a multitude of information such as services performed, mileage, dates, receipts, photos, and notes. By doing this a log is created where information on your car can be easily organized. Furthermore, the app provides default maintenance schedules based on vehicle fuel type, which users can customize to keep themselves accountable. Maintenance Buddy will be a tool that will consistently help in caring for and maintaining your vehicle.

2. Functional Requirements

2.1 user_registration

Type: Functional

Summary: The system should allow users to register an account using email and password.

2.2 user_login

Type: Functional

Summary: Users should be able to securely log in to their account.

2.3 user_logout

Type: Functional

Summary: Users should be able to log out of their account.

2.4 reset_password

Type: Functional

Summary: The system should allow users to change their password.

2.5 add_vehicle

Type: Functional

Summary: The system should allow users to add a car to their profile with the correct year, make, model, fuel type (petrol, diesel, hybrid, or electric), and yearly mileage estimate. Upon adding a vehicle, the system should present a default maintenance schedule based on the fuel type, which the user can customize by adjusting service intervals.

2.6 vehicle_edit

Type: Functional

Summary: Users should be able to edit vehicle information that is on their profiles.

2.7 vehicle_delete

Type: Functional

Summary: Users should be able to delete vehicles that have been added to their profile.

2.8 create_service_event

Type: Functional

Summary: Users should be able to create a service event, with the date and other service details.

2.9 edit_service_event

Type: Functional

Summary: Users should be able to edit previously created service events.

2.10 delete_service_event

Type: Functional

Summary: Users should be able to delete service events from their vehicle history.

2.11 upload_photos

Type: Functional

Summary: Users should be able to upload up to 3 photos (PNG, JPEG) to a service event. Each photo is limited to 10MB.

2.12 upload_docs

Type: Functional

Summary: Users should be able to upload text-style documents, including PDF and TXT, to a service event. Service events should be limited to 1 document attachment. Each document is limited to 10MB.

2.13 set_mileage_rate

Type: Functional

Summary: When adding a vehicle, the app should prompt users to estimate their yearly mileage for that vehicle. Default options should be provided. The app should estimate each vehicle's mileage independently based on its provided mileage rate.

2.14 update_vehicle_mileage

Type: Functional

Summary: Upon completion of a service, the app should encourage the user to update their vehicle's current mileage so that cumulative estimation error may be reduced.

2.15 maintenance_notifications

Type: Functional

Summary: The system should send a notification to the user when it is time for vehicle maintenance by referencing the estimated current mileage.

2.16 export_summary_report

Type: Functional

Summary: Users should be able to export a condensed maintenance summary in PDF format. Users may export a summary for a single vehicle or for all vehicles combined.

2.17 summary_report_content

Type: Constraint

Summary: The summary report should be limited to the most recent 30 services/3 pages and be text-only with a limit of 400 characters per service.

2.18 export_full_report

Type: Functional

Summary: Users should be able to export the complete and unaltered maintenance history as a ZIP archive (one archive per vehicle, images included), organized as one folder per service.

2.19 notification_center

Type: Functional

Summary: Users should be able to view and manage notifications in a dedicated notification center. Notifications should be separated into unread and read sections. Users should be able to mark individual notifications as read or mark all as read. An unread count badge should be displayed on the notifications tab.

2.20 required_services_management

Type: Functional

Summary: Users should be able to view and manage their vehicle's required services on a dedicated screen. Users should be able to edit the mileage interval for any service and enable or disable individual services.

2.21 multi_vehicle_dashboard

Type: Functional

Summary: The main dashboard should display all of the user's vehicles and show upcoming appointments across all vehicles. Users should be able to navigate to any vehicle or add a new vehicle from the dashboard.

2.22 unit_preference

Type: Functional

Summary: Users should be able to toggle between imperial (miles) and metric (kilometers) units in the settings screen.

2.23 skip_first_reminder

Type: Functional

Summary: When adding a vehicle, users should be able to skip the first maintenance reminder for individual services that have already been recently performed.

3. Interface Requirements

3.1 organized_interface

Type: Constraint

Summary: The user should be able to easily interact with and understand the app.

3.2 mobile_interface

Type: Constraint

Summary: The interface should be mobile friendly.

3.3 camera_integration

Type: Functional

Summary: The system should be able to access and use the camera in order to take pictures.

3.4 photo_gallery_access

Type: Functional

Summary: The system should be able to have access to the gallery in order to upload pictures.

3.5 file_storage_access

Type: Functional

Summary: The system should have access to the files in system storage for the purpose of uploading them to the software.

3.6 notification_interface

Type: Constraint

Summary: The system should interface with the mobile operating system to send push notifications for maintenance reminders.

3.7 android_support

Type: Constraint

Summary: The system should work for android devices.

3.8 os_update_compatibility_risk

Type: Risk

Summary: Future operating system updates may cause compatibility issues that affect application functionality.

4. Performance Requirements

4.1 upload_speed

Type: Constraint

Summary: Photos and files should be uploaded in a timely manner.

4.2 upload_size

Type: Constraint

Summary: Uploaded photos and files should be limited in size. All attachments (photos and documents) are limited to 10MB each.

4.3 notification_speed

Type: Constraint

Summary: Notifications for vehicle maintenance reminders should be dispersed at the proper intervals.

4.4 scalability

Type: Constraint

Summary: The system should handle increasing numbers of users without significant performance degradation.

5. Non-Functional Requirements

5.1 software_availability

Type: Constraint

Summary: The system should be available to run on Android devices and requires an active internet connection for full functionality.

5.2 software_maintainability

Type: Constraint

Summary: The system should be structured in a way that allows future updates and feature additions without major redesign.

5.3 software_security

Type: Constraint

Summary: The system must be secure and protect user accounts, vehicle data, and uploaded documents from unauthorized access through proper authentication and encryption measures.

5.4 software_reliability

Type: Constraint

Summary: The system must be reliable at all times without crashing.

5.5 data_backup

Type: Constraint

Summary: The system should be able to back up data regularly in case of data loss.

5.6 data_loss

Type: Risk

Summary: Unexpected system failures may result in loss of data.
